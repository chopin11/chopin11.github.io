CRL是什么

我们知道，数字证书一般可用于服务器和客户端，在服务器端是作为站点证书，在客户端是作为用户证书。在使用证书的过程中，可能会遇到下边的情况：

1. 服务器证书的私钥被攻击者窃取，那攻击者就可以随意伪造对应网站的身份，我们该怎么办？

2. 作为用户证书，员工不慎将证书遗失，该怎么挂失证书？

3. 作为用户证书，如何来限制离职员工证书的正常使用？

4. 由于某些原因，CA 机构错误签发了一张证书，如何取消该证书？

针对上述类似的情况，PKI 提供了一种机制 证书吊销列表，缩写为 CRL 。简单的说，CRL 其实就是一个文件，记录了所有已经被吊销的证书列表，由证书颁发机构 CA 来维护。

当用户使用浏览器访问网站时，除了要验证服务器证书的身份，还要下载对应 CRL 文件进行校验，确认证书是否已被吊销，对于已吊销的证书视为无效；

针对上述的 4 个场景问题，都可以通过证书吊销列表的方式进行解决，因为有了 CRL，浏览器就可以将异常的证书视为无效，从而最大程度的规避了安全隐患。

工作原理

CRL 本质是一个文件，与证书类似，是一个结构化的文件，CA 机构将所有吊销的证书集成在一个文件中。其实它相当于一个黑名单，包含了所有被吊销证书的序列号和吊销原因。

接下来，我们先了解一下 CRL 工作原理，本质上就是要弄清楚证书校验方（浏览器）是如何使用它进行证书校验的。现在我们要换个角度，站在浏览器的角度来思考如何校验

1. 浏览器接收到服务器证书后，首先通过证书链校验证书是否可信，如果校验成功，接着执行证书吊销的校验；

2. 浏览器从服务器证书中找到 CRL 分发点，其实就是证书吊销列表的下载链接；

3. 浏览器将 CRL 文件下载并缓存，由于 CRL 有数字签名进行保护，浏览器需要找到对应公钥进行验签。首先在 CRL 文件中找出签发者，从而找到完整的证书链，最终完成签名验签；

4. 签名验签成功后，根据 x509 标准解析 CRL 文件，生成结构化的数据；

5. 从解析数据中查找，如果证书的序列号存在于 CRL 中，则说明证书已被吊销；

还有一些其它需要注意和了解的内容，总结如下

  • 为了加快握手速度，某些浏览器不校验签名，直接查询从而完成 CRL 校验；
  • 由于 CRL 文件可能非常大，浏览器下载时非常耗时，为避免频繁下载，浏览器一般会定期缓存，减少对 CRL 请求；
  • 在 CRL 校验完成之前，TLS 握手过程是阻塞的，会延长握手时间；
  • 如果 CRL 下载超时，浏览器一般会忽略检查证书吊销状态；
  • 理论上服务器证书和中间 CA 证书都需要进行 CRL 校验，但大部分浏览器从效率考虑，一般只校验服务器证书的吊销状态；

问题思考

在理解上述原理时，为了简化过程便于理解，有的步骤描述得有点抽象，为了更详细和深入的理解，扩展了一些值得深入思考的点

问：证书吊销列表 CRL 为什么需要数字签名？

答：为了避免 CRL 被篡改，CA 对 CRL 进行数字签名。需要注意的是，CA 机构签发服务器证书的私钥和签发 CRL 私钥可以是不同的，甚至可以委托另一家 CA 机构签发 CRL。

如何校验 CRL 签名？

答：与服务器证书校验一样，浏览器需要通过完整证书链来对 CRL 验签。CRL 文件一般提供签发者的信息，浏览器就可以从 CRL 文件中依次迭代找出证书链，构建完整证书链，验证 CRL 的合法性。

问：浏览器如何找到 CRL 文件的？

答：CRL 由 CA 机构发布，该文件在互联网上有一个 URL 地址，该地址称为 CRL 分发点，以 HTTP 形式提供。浏览器是如何知道 CRL 分发点呢？如果 CA 支持 CRL 功能，就会在服务器证书扩展中添加 CRL 分发点。

问：证书过期和证书吊销的区别？

答：这是两个不同的概念，证书过期并不代表证书吊销，所有过期的证书肯定是无效的，但没有过期的证书可能已经被吊销，同样是无效的。证书吊销，可以理解为颁发者提供一种撤销证书的能力。

CRL格式

理解了 CRL 原理之后，再来学习 CRL 格式就会更容易些。接下来从协议的角度了解 CRL 的内容结构

（一）CertificateList 结构

类似于证书形式的发布，发布者会对其进行签名。

  • signatureAlgorithm 指明签名算法
  • signatureValue 表示签名
  • tbsCertList 主要包含签发者、签发日期、吊销证书列表等信息

（二）tbsCertList 结构

  • issuer 指定 CRL 签发者；
  • thisUpdate 和 nextUpdate CA 用来告知浏览器 CRL 本次更新时间和下次更新时间，浏览器可以根据这两个时间点缓存，从而减少对 CRL 请求；
  • revokedCertificates 所有被吊销的证书的序列号都包含在该字段中，也包含吊销时间和吊销原因；
  • crlExtensions 常见的扩展包括 Authority Key Identifier、CRL Number、Authority Informationn Access，一般是从这三个字段来获取签发者证书的。

（三）revokedCertificates结构

包含了被吊销服务器证书的信息，主要包括三部分

  • userCertificate 证书的序列号
  • revocationDate 证书被吊销的时间
  • crlEntryExtensions 扩展选项，主要包括 Reason Code 吊销原因、Invalidity Date 吊销时间、Certificate Issuer签发者。

关于 CRL 格式先了解这么多就可以了。

存在问题

到此，我们可以理解 CRL 功能是非常重要的，但是它确实也存在着很多问题，我们来一起梳理一下

1. CRL 维护的复杂性

对于 CA 机构来说，随着签发证书越多，被吊销的证书就会越来越多，CRL 文件就会变得很大，浏览器下载速度就会很慢，影响整体握手速度。

2. CRL 更新的实时性

一般情况下，CA 结构收到证书吊销请求后，并不会立即更新 CRL，即便立即更新，由于浏览器对 CRL 是周期更新缓存的，被吊销的证书不能够实时地被浏览器校验识别出来，存在安全风险。

3. CRL 校验的复杂度

校验时，必须下载完整的 CRL 文件，即使只为了校验一张证书，也需要进行复杂的解析操作；另外，除了服务器证书，中间 CA 证书也需要进行校验，极大增加了浏览器校验的复杂度。

如果一个 CRL 文件没有完成下载，该如何确认该证书的吊销状态呢？终止握手过程，会影响用户体验，一般浏览器选择跳过 CRL 验证，同样存在安全风险。

4. 影响通信效率

对于浏览器来说，检查证书的吊销状态是个耗时的操作，只有校验完毕才能进行后续的握手，是一个阻塞操作，严重影响通信的效率。

由于存在这么多问题，目前浏览器厂商和 CA 机构都逐渐弱化 CRL 的使用：

  • Let’s Encrypt 签发的证书已经不包含 CRL 分发点，也就是不提供 CRL 服务；
  • Firefox 28 以后的版本，将不会校验 CRL，使用 OCSP 机制代替；
  • Chrome 19 以后的版本也不会校验 CRL，但也不使用 OCSP 机制；

由于历史原因和当时互联网环境，设计的 CRL 功能存在一系列的问题，目前已经逐渐被 OCSP 淘汰。但这并不代表学习 CRL 毫无意义，对于某些应用还是有使用场景的，同时也可以进一步加深对于密码学的理解。
